알고리즘 문제를 풀다 보면 "똑같은 로직인데 왜 이 쪽이 더 느리지?" 싶은 순간이 종종 온다.
그 원인 중 하나가 **포인터 앨리어싱(pointer aliasing)** 이다.

## 앨리어싱이 뭐길래

두 포인터가 같은 메모리 영역을 가리킬 수 있다는 가능성 자체를 가리킨다.
C/C++ 표준은 `char*` 를 제외하면 같은 타입 간에는 앨리어싱이 허용되기 때문에,
컴파일러는 "혹시 겹치면 어쩌지?" 를 고려해서 **메모리에 더 자주 접근**한다.

```cpp
void add(int* a, int* b, int* c, int n) {
    for (int i = 0; i < n; ++i) {
        c[i] = a[i] + b[i];
    }
}
```

언뜻 보면 단순해 보이지만, 컴파일러는 `c` 가 `a` 또는 `b` 와 겹칠 수 있다고 가정한다.
그 결과 SIMD 벡터화·루프 언롤링을 보수적으로 포기할 가능성이 생긴다.

## `__restrict` 키워드

GCC/Clang/MSVC 모두 비표준 확장으로 `__restrict` (혹은 `__restrict__`) 를 지원한다.
"이 포인터가 가리키는 메모리는 다른 포인터와 겹치지 않는다" 를 컴파일러에게 약속하는 것이다.

```cpp
void add(int* __restrict a, int* __restrict b, int* __restrict c, int n) {
    for (int i = 0; i < n; ++i) {
        c[i] = a[i] + b[i];
    }
}
```

같은 로직인데, 컴파일러가 벡터화·루프 언롤링을 적극적으로 적용할 수 있다.

## 실측

`n = 1 << 24`, `-O2 -march=native` 기준 간단 벤치:

| 버전 | 시간 |
|---|---|
| 일반 포인터 | 41.2 ms |
| `__restrict` | 14.8 ms |

2~3배 차이가 흔히 난다. 물론 워크로드와 컴파일러 버전에 따라 편차가 크다.

## 주의할 점

- 약속을 **어기면 UB** 다. 겹치는 메모리를 넘기면 컴파일러는 "안 겹친다" 는 가정으로 옵티마이즈해버린다.
- `std::vector<int>::data()` 는 `__restrict` 가 아니다. STL 은 규약상 이걸 약속해줄 수 없다.
- 단위 테스트 커버리지가 없으면 쓰지 마라. 실수로 aliased 호출을 허용하면 흔적 없이 버그가 생긴다.

## 알고리즘 문제에서의 판단

- 짧은 루프에서는 체감이 거의 없다.
- 행렬 연산·컨볼루션처럼 **길고 단순한 루프**가 있으면 우선 `__restrict` 를 떠올리자.
- 그 전에 `ios_base::sync_with_stdio(false)` 와 `cin.tie(nullptr)` 가 이미 적용돼 있는지부터 확인.

---

포인터는 단순한 주소값이지만, 그 뒤에 깔린 "혹시"라는 가정이 성능을 갉아먹는다.
그 가정을 명시적으로 깎아내는 것만으로도 꽤 차이가 난다.
